二、相应安全机制简要描述

分布式公证人Demo安全机制主要由分布式密钥、HTLC原子互换、阈值签名及零知识证明等构成。

（1）分布式密钥技术

   在分布式公证人Demo设计中，无论主链合约和侧链合约均是由公证人部署生成。用户向侧链（Spectrum）转移token,需要在主链（以太坊）合约中锁定token.如果某个公证人完全掌握这个帐户的私钥，存在相当大的安全隐患。我们采用并设计了相应的分布式密钥技术，由多个公证人联合掌握合约帐户的私钥，每个公证人只生成并保存私钥的一部分（即私钥片），彼此之间并不发生私钥片的传递和组装。任何人都没有完整的私钥，对账户资产的解锁和转移需要得到大多数（阈值）公证人的确认才能完成，有效的降低了私钥泄露以及多个公证人(小于阈值)联合作恶的风险。

（2）HTLC原子互换

 在分布式公证人Demo设计中，如果用户发送token进主链合约进行锁定，而公证人没有采取相应的映射token生成操作，或当用户想解锁侧链token回主链账户时，公证人不作为，将对用户造成损失。我们采用并设计了HTLC原子互换操作，将主链锁定token与侧链生成映射token等环节构建成原子操作，通过密码和时间保证跨链交易要么都成功，要么都失败，不会造成用户的损失。

（3）阈值签名

 在资产还原回主链的过程中，如果需要全部公证人签名确认的话，则假如某个公证人不在线或者不作为，将导致交易失败，资产无法回到主链。我们采用并设计了阈值签名技术，由m个公证人组成的签名组中，只要大于阈值t（t≤m）个公证人确认签名，即可完成交易的确认，有效的提高了交易的成功率和效率，也防范了个别公证人的不作为攻击。

（4）零知识证明

  零知识证明可以在不泄露个人信息的前提下证明自己是某信息的合法拥有者以及证明自己的身份。在分布式密钥以及阈值签名过程中，需要对同态公钥的正确性、私钥片的正确拥有性、签名片的正确拥有性以及公证人身份等重要信息进行验证。为了加强流程中的交互安全以及防止恶意公证人仿照或破坏交易，我们lockin和lockout多个步骤中使用了Dlog proof、Correct key proof、Homomorphic EIGamal proof等零知识证明实现对秘密信息的隐藏及非交互式验证，任一环节通不过，则交易自动失败，有效的保证交易的安全性。

  此外，在Demo具体设计中，我们也使用了椭圆曲线加密、VSS多方联合秘密共享、共享对话、Paillier同态加密、陷门承诺等加密技术，限于篇幅不过多介绍。后期在公证人选拔中我们还将采用公证人组约束、惩罚机制、退出机制、竞争机制等提高整个跨链系统的安全性。
